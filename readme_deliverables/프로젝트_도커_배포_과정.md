# 현재 프로젝트 도커 배포 과정 정리

## 1. 문서 목적

이 문서는 현재 프로젝트의 구조와 설정 파일을 기준으로, `Spring Boot 백엔드 + Vue 프론트엔드 + PostgreSQL` 조합을 Docker 환경에 배포하는 과정을 정리한 문서다.

현재 저장소를 확인한 결과:

- 백엔드 프로젝트는 [readme](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme) 에 존재한다.
- 프론트엔드 프로젝트는 [readme_vue](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme_vue) 에 존재한다.
- 배포 구조는 산출물 문서에서 `Docker Compose + Nginx` 기반을 목표로 하고 있다.
- 하지만 현재 저장소에는 실제 배포용 `Dockerfile`, `docker-compose.yml`, `nginx.conf` 파일은 아직 포함되어 있지 않다.

즉, 이 문서는 **현재 코드 기준의 실제 배포 절차 정리 + 필요한 도커 배포 구성 가이드** 성격을 가진다.

## 3. 배포 전 준비해야 할 항목

- Docker Compose 사용 가능 환경 준비
- 배포 서버 포트 개방

- 실제 비밀값은 Git에 올리지 않는다.
- 서버용 `.env` 파일을 별도로 둔다.
- 결제 키, OAuth 시크릿, JWT 시크릿은 운영용 값으로 교체한다.

## 4. 현재 코드 기준 배포 시 수정이 필요한 부분

현재 코드를 그대로 도커에 올리면 바로 부딪히는 부분이 있다.

### 4.1 DB 접속 주소 변경 필요

현재 백엔드 설정:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5420/readme
```

Docker Compose 환경에서는 `localhost`가 백엔드 컨테이너 자기 자신을 가리키므로 동작하지 않는다.

배포 시에는 다음과 같이 바꿔야 한다.

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST:postgres}:${DB_PORT:5432}/${DB_NAME:readme}
```

또는 환경변수 하나로 통째로 관리해도 된다.

```yaml
spring:
  datasource:
    url: ${SPRING_DATASOURCE_URL}
```

예시:

```env
SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/readme
```

### 4.2 프론트 API 주소 변경 필요

현재 프론트 설정:

```js
baseURL: import.meta.env.VITE_API_BASE_URL || 'http://localhost:8202'
```

배포 시에는 다음 두 방식 중 하나로 맞춰야 한다.

- 방법 1: 프론트에서 `VITE_API_BASE_URL=https://your-domain.com/api` 사용
- 방법 2: Nginx가 `/api` 요청을 백엔드로 프록시하고, 프론트는 상대경로 기반으로 호출

실무적으로는 두 번째 방식이 더 깔끔하다.

### 4.3 업로드 파일 볼륨 마운트 필요

현재 파일 업로드는 `uploads` 디렉터리를 사용한다.

- [WebMvcConfig.java](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme/src/main/java/com/bookstore/shop/readme/config/WebMvcConfig.java) 에서 `/uploads/**` 정적 경로 제공
- [FileStorageService.java](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme/src/main/java/com/bookstore/shop/readme/service/FileStorageService.java) 에서 파일 저장

도커 배포 시 컨테이너가 재생성되어도 이미지가 남도록 볼륨을 반드시 연결해야 한다.

예시:

```yaml
volumes:
  - ./data/uploads:/app/uploads
```

## 5. 권장 도커 배포 구성

현재 프로젝트 기준 권장 배포 구조는 아래와 같다.

### 5.1 서비스 구성

- `frontend`
  - Vue 앱 빌드 후 Nginx에서 정적 서빙
- `backend`
  - Spring Boot JAR 실행
- `postgres`
  - PostgreSQL DB
- `nginx`
  - 외부 요청 진입점
  - 프론트 정적 파일 서빙 또는 frontend 컨테이너 프록시
  - `/api`는 backend로 프록시
  - `/uploads`는 backend 또는 파일 저장 경로로 연결

### 5.2 권장 네트워크 흐름

```text
사용자 브라우저
  -> Nginx
    -> /            프론트 정적 파일
    -> /api         Spring Boot API
    -> /uploads     업로드 파일 접근

Spring Boot
  -> PostgreSQL
```

## 6. 권장 배포 파일 예시

아래는 현재 프로젝트를 배포하기 위한 대표적인 파일 구성 예시다.

### 6.1 백엔드 Dockerfile 예시

위치 예시: `readme/Dockerfile`

```dockerfile
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /app

COPY . .
RUN chmod +x gradlew
RUN ./gradlew clean bootJar -x test

FROM eclipse-temurin:21-jre
WORKDIR /app

COPY --from=builder /app/build/libs/*.jar app.jar

EXPOSE 8202

ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 6.2 프론트 Dockerfile 예시

위치 예시: `readme_vue/Dockerfile`

```dockerfile
FROM node:22-alpine AS builder
WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .
RUN npm run build

FROM nginx:stable-alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
```

### 6.3 docker-compose.yml 예시

프로젝트 루트에 두는 것을 권장한다.

```yaml
version: "3.9"

services:
  postgres:
    image: postgres:16
    container_name: readme-postgres
    environment:
      POSTGRES_DB: readme
      POSTGRES_USER: ${DB_USERNAME}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5420:5432"
    volumes:
      - ./data/postgres:/var/lib/postgresql/data

  backend:
    build:
      context: ./readme
    container_name: readme-backend
    depends_on:
      - postgres
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/readme
      DB_USERNAME: ${DB_USERNAME}
      DB_PASSWORD: ${DB_PASSWORD}
      JWT_SECRET: ${JWT_SECRET}
      GOOGLE_CLIENT_ID: ${GOOGLE_CLIENT_ID}
      GOOGLE_CLIENT_SECRET: ${GOOGLE_CLIENT_SECRET}
      KAKAO_CLIENT_ID: ${KAKAO_CLIENT_ID}
      KAKAO_CLIENT_SECRET: ${KAKAO_CLIENT_SECRET}
      APP_BASE_URL: ${APP_BASE_URL}
      FRONT_OAUTH_REDIRECT_URI: ${FRONT_OAUTH_REDIRECT_URI}
      CORS_ALLOWED_ORIGINS: ${CORS_ALLOWED_ORIGINS}
      FILE_UPLOAD_DIR: /app/uploads
      TOSS_CLIENT_KEY: ${TOSS_CLIENT_KEY}
      TOSS_SECRET_KEY: ${TOSS_SECRET_KEY}
      KAKAO_SECRET_KEY: ${KAKAO_SECRET_KEY}
      NAVER_PAY_CLIENT_ID: ${NAVER_PAY_CLIENT_ID}
      NAVER_PAY_CLIENT_SECRET: ${NAVER_PAY_CLIENT_SECRET}
      NAVER_PAY_CHAIN_ID: ${NAVER_PAY_CHAIN_ID}
      NAVER_PAY_MERCHANT_ID: ${NAVER_PAY_MERCHANT_ID}
    ports:
      - "8202:8202"
    volumes:
      - ./data/uploads:/app/uploads

  frontend:
    build:
      context: ./readme_vue
    container_name: readme-frontend
    depends_on:
      - backend
    ports:
      - "80:80"
```

### 6.4 Nginx 설정 예시

프론트 컨테이너 또는 별도 Nginx 컨테이너에서 사용할 수 있다.

```nginx
server {
    listen 80;
    server_name your-domain.com;

    root /usr/share/nginx/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://backend:8202/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /uploads/ {
        proxy_pass http://backend:8202/uploads/;
    }
}
```

## 8. 운영 중 체크 포인트

### 8.1 CORS

현재 백엔드는 `app.cors.allowed-origins` 값을 기준으로 CORS를 처리한다.

- [WebMvcConfig.java](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme/src/main/java/com/bookstore/shop/readme/config/WebMvcConfig.java)

배포 시에는 다음처럼 도메인을 정확히 지정해야 한다.

```env
CORS_ALLOWED_ORIGINS=https://your-domain.com,https://www.your-domain.com
```

### 8.2 OAuth Redirect URI

Google, Kakao OAuth는 운영 도메인 기준으로 Redirect URI를 다시 등록해야 한다.

로컬 주소가 그대로 들어 있으면 운영 환경에서 로그인에 실패한다.

### 8.3 결제 연동 키

현재 `.env`에는 테스트용/개발용 결제 키가 섞여 있을 가능성이 있으므로 운영 배포 시 반드시 교체해야 한다.

특히 아래 항목은 운영용 값 사용 여부를 점검해야 한다.

- Toss Payments
- Kakao Pay
- Naver Pay

### 8.4 파일 업로드 유지

업로드 디렉터리는 컨테이너 내부 경로만 사용하면 재배포 시 사라질 수 있다.

반드시 볼륨 마운트를 설정한다.

```yaml
volumes:
  - ./data/uploads:/app/uploads
```

### 8.5 HTTPS

산출물 문서에서도 HTTPS 적용 예정이 명시되어 있다.

- [프로젝트 개요서.md](/C:/Users/ez/Claude/git/hwandroid921/teamProject/dev/readme_deliverables/최종/프로젝트%20개요서.md)

운영 환경에서는 다음을 권장한다.

- Nginx + SSL 인증서 적용
- OAuth/결제 콜백 URL 모두 `https://` 사용
- 쿠키/인증 헤더 정책 점검

## 9. 현재 프로젝트 기준 배포 리스크

현재 상태에서 바로 도커 배포하려면 아래 리스크를 먼저 해결해야 한다.

### 9.1 실제 배포 파일 부재

현재 저장소에는 아래 파일이 없다.

- `readme/Dockerfile`
- `readme_vue/Dockerfile`
- `docker-compose.yml`
- `nginx.conf`

즉, 문서상 배포 구조는 정의되어 있지만 실제 배포 자산은 아직 미완성 상태다.

### 9.2 DB URL 하드코딩

현재 DB 접속 주소가 `localhost:5420` 기준이므로, Compose 네트워크 환경에서는 수정이 필요하다.

### 9.3 프론트-백엔드 URL 정리 필요

프론트는 기본적으로 `http://localhost:8202`를 바라보므로 운영 도메인 또는 `/api` 상대경로 기준으로 정리하는 것이 좋다.

### 9.4 민감정보 관리

현재 `.env`에 민감한 값이 포함되어 있으므로 운영 전에는 반드시 재발급 및 분리가 필요하다.

## 10. 최종 정리

현재 프로젝트의 도커 배포 흐름은 아래처럼 이해하면 된다.

1. Vue 프론트엔드를 빌드한다.
2. Spring Boot 백엔드를 JAR로 빌드한다.
3. PostgreSQL 컨테이너를 준비한다.
4. Docker Compose로 프론트, 백엔드, DB를 함께 올린다.
5. Nginx가 정적 파일을 서빙하고 `/api` 요청을 백엔드로 전달한다.
6. 업로드 파일은 볼륨으로 분리해 영속 저장한다.
7. CORS, OAuth, 결제 콜백 URL을 운영 도메인 기준으로 맞춘다.

즉, **현재 프로젝트는 도커 배포를 고려한 설계와 환경변수 구조는 갖춰져 있지만, 실제 배포용 파일은 아직 작성되지 않은 상태**라고 정리할 수 있다.

7. 업로드 볼륨 및 OAuth/결제 리다이렉트 주소 점검
8. 서버에서 `docker compose up -d --build`로 실제 검증

- 실제 프로젝트에 맞는 `docker-compose.yml` 작성
- 백엔드/프론트 `Dockerfile` 생성
- `nginx.conf` 생성
- 배포용 `.env.example` 파일 생성
