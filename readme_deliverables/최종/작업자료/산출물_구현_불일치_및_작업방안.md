## 🔴 산출물과 구현 내용이 다른 항목 (즉시 확인 필요)

### 1. QnA 상태 관리 방식 불일치

**산출물(기능명세서) 내용**
- `qna_status` ENUM: `WAITING → PROCESSING → COMPLETE`
- FA-040: 관리자 QnA 상태 변경 API 필요

**실제 구현**
- QnA 엔티티에 status 필드 없음
- `answer == null` → 미답변 / `answer != null` → 답변 완료로만 구분
- PROCESSING 상태 없음

**작업 방안 (선택)**

방안 A (권장, D-4 현실적): answer 필드로 대체 처리
- `answer == null` → WAITING, `answer != null` → COMPLETE 로 프론트에서 표시
- QnAResponse에 `status` 필드를 computed property로 추가

```java
// QnAResponse.java에 추가
public String getStatus() {
    return this.answer != null ? "COMPLETE" : "WAITING";
}
```

방안 B (산출물 완전 준수, 시간 소요): QnA 엔티티에 QnaStatus ENUM 추가
```java
// QnA.java에 추가 (DB 마이그레이션 필요)
@Enumerated(EnumType.STRING)
@Builder.Default
private QnaStatus qnaStatus = QnaStatus.WAITING;
```

### 2. 관리자 QnA 상태 변경 API 누락

**산출물**: `PATCH /admin/qna/{id}` — 상태 WAITING → PROCESSING → COMPLETE 변경

**실제 구현**: 없음 (답변 등록 시 자동으로 완료 처리됨)

**작업 방안 (방안 A 선택 시)**
AdminApiController에 아래 엔드포인트 추가 불필요 (answer 등록 = 완료 처리)

**작업 방안 (방안 B 선택 시)**
```java
// AdminApiController
@PatchMapping("/qnas/{qnaId}/status")
public ResponseEntity<String> updateQnAStatus(
        @PathVariable Long qnaId,
        @RequestParam String status) {
    return qnaService.updateQnAStatus(qnaId, status);
}
```

### 3. 장바구니 is_checked 토글 API 누락

**산출물**: REQ-C-004 — `is_checked` 컬럼으로 주문 포함/제외 처리

**실제 구현**
- `CartItem` 엔티티에 `isChecked` 필드는 있음
- `CartUpdateRequest`에 checked 파라미터 없음
- 토글 API 엔드포인트 없음

**작업 방안**
```java
// CartUpdateRequest.java 수정
public record CartUpdateRequest(
    int quantity,
    Boolean isChecked  // 추가: null이면 수량만 변경
) {}

// CartApiController에 추가
@PatchMapping("/{cartItemId}/check")
public ResponseEntity<String> toggleCartItemCheck(
        @PathVariable Long cartItemId,
        @RequestBody Map<String, Boolean> body,
        @AuthenticationPrincipal CustomUserDetails user) {
    cartService.toggleCheck(cartItemId, body.get("isChecked"), user.getMemberId());
    return ResponseEntity.ok("선택 상태가 변경되었습니다.");
}
```

### 4. OAuth provider 불일치 예외 처리 미구현

**산출물**: REQ-AUTH-005 — 동일 이메일로 다른 provider 로그인 시도 시 오류 반환

**실제 구현**
- `OAuthMemberService.findOrCreate()`에서 동일 이메일 다른 provider 체크 없음
- 카카오로 가입한 사용자가 구글 로그인 시도 시 자동으로 병합됨

**작업 방안**
```java
// OAuthMemberService.findOrCreate() 수정
Optional<Member> existingByEmail = memberRepository.findByEmail(email);
if (existingByEmail.isPresent() && existingByEmail.get().getProvider() != provider) {
    throw new OAuthProviderMismatchException(
        existingByEmail.get().getProvider() + "로 이미 가입된 이메일입니다.");
}
```
GlobalExceptionHandler에 OAuthProviderMismatchException → 409 처리 추가

### 5. 공지사항 조회수 (view_count) 미구현

**산출물**: REQ-N-002 — 공지사항 상세 조회 시 view_count 자동 증가

**실제 구현**
- Notice 엔티티에 `viewCount` 필드 없음
- NoticeService.getNotice() 조회만 처리

**작업 방안**
```java
// Notice.java에 추가
@Column(nullable = false)
@Builder.Default
private int viewCount = 0;

// NoticeService.getNotice() 수정
notice.setViewCount(notice.getViewCount() + 1);
```

### 6. 미작성 리뷰 목록 조회 (REQ-M-020) 미구현

**산출물**: 구매 완료 후 리뷰 미작성 상품 목록 조회 (`is_reviewed = false` 기준)

**실제 구현**
- `OrderItem.isReviewed` 필드는 존재하나 항상 false (갱신 로직 없음)
- 리뷰 작성 시 isReviewed → true 로 변경하는 로직 없음

**작업 방안**
```java
// ReviewService.createReview() 수정 — 리뷰 등록 후 OrderItem.isReviewed = true
orderItemRepository
    .findByOrderMemberIdAndProductId(memberId, req.productId())
    .ifPresent(item -> item.setIsReviewed(true));

// MyPageApiController에 추가
@GetMapping("/me/reviews/pending")
public ResponseEntity<List<OrderItemPendingReviewResponse>> getPendingReviews(
        @AuthenticationPrincipal CustomUserDetails user) {
    return reviewService.getPendingReviews(user.getMemberId());
}
```
