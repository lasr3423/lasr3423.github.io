### 1. 회원 주문 취소 환불 처리 불일치 (해결 완료)

#### 기존 현상

- 프론트 마이페이지 주문 목록에서는 아래 상태를 취소 가능으로 판단합니다.
  - `PAYMENT_PENDING`
  - `PENDING`
  - `PAYED`
- 하지만 백엔드 주문 취소 로직은 `PAYED`일 때만 `paymentService.cancelPayment()`를 호출합니다.
- 현재 결제 완료 후 주문 상태는 `PENDING`으로 바뀌므로, 실제 서비스에서는 **결제가 완료된 주문이 `PENDING` 상태일 가능성이 높습니다.**

#### 문제점

- 사용자는 주문 취소가 되었다고 보지만 실제 PG 취소가 수행되지 않을 수 있습니다.
- 즉, **주문 상태와 결제 상태가 분리되어 꼬일 위험**이 있습니다.

- 주문 상태가 아니라 `Payment` 존재 여부와 `PaymentStatus`를 기준으로 환불/취소를 판별하도록 변경했습니다.
- `PAYED` 레거시 상태에만 묶여 있던 취소 로직을 제거했습니다.
- 주문 취소 정책 기준을 아래와 같이 정리하고 코드에 반영했습니다.

```text

PENDING/PAYED   : 결제 전액 환불 후 주문 취소

DELIVERED       : 주문 취소 불가 (별도 반품/교환 프로세스로 분리)
```

- 참고 정책:
  - 교환비: 3,000원
  - 반품비: 6,000원

- 현재 이번 수정 범위에서는 **주문 취소 시 반품비 6,000원 차감 환불**까지 반영했고,

- `PAYMENT_PENDING`, `PENDING`, `PAYED`, `APPROVAL`, `DELIVERING` 주문은 취소 가능 상태로 통일했습니다.
- `DELIVERING` 상태 취소 시 반품비 6,000원을 차감한 환불 금액을 계산합니다.

### 2. 결제 완료 후 배송 생성 시점 불일치 (해결 완료)

#### 기존 현상

- 기존에는 결제 완료 시점에 바로 배송 레코드를 생성하는 로직이 있었습니다.
- 하지만 현재 주문 상태 설계는 다음 흐름을 따릅니다.

```text
PAYMENT_PENDING -> PENDING -> APPROVAL -> DELIVERING -> DELIVERED
```

- 즉 `PENDING`은 결제 완료 후 관리자 승인 대기 상태입니다.

#### 문제점

- `PENDING`인데 이미 배송 레코드가 생성되면 상태 의미가 흐려집니다.
- “결제 확인”과 “배송 승인”이 분리된 운영 정책과 실제 코드가 맞지 않습니다.

#### 적용한 수정

- `PaymentService.confirmToss(...)`에서 결제 완료 후 배송 생성 로직을 제거했습니다.
- `PaymentService.approvePayment(...)`에서도 결제 완료 후 배송 생성 로직을 제거했습니다.
- 이제 결제 완료 시점에는 주문 상태만 `PENDING`으로 변경되고, 배송 레코드는 생성되지 않습니다.
- 배송 레코드는 `OrderService.applyOrderStatusChange(...)`에서 주문이 `APPROVAL`로 전환될 때만 `ensureReadyDelivery(...)`를 통해 생성됩니다.

#### 현재 동작 기준

- 배송 레코드는 결제 완료 시점이 아니라 `APPROVAL` 전환 시점에만 생성되도록 통일하는 것이 맞습니다.
- 결제 서비스는 결제 승인과 주문 상태 `PENDING` 전환까지만 담당합니다.
- 주문 승인 로직만 배송 생성을 담당하므로 운영 정책과 코드 흐름이 일치합니다.
