1. 표지
시간: 30초~1분

“안녕하세요. 저희는 Vue 3, Spring Boot, PostgreSQL을 기반으로 온라인 서점 웹 서비스를 구현한 팀입니다.
이번 발표에서는 프로젝트를 기획하게 된 배경부터 시스템 구조, 핵심 기능, 구현 과정에서의 어려움과 해결 방법까지 설명드리고, 마지막에는 실제 구현 화면을 시연하겠습니다.”

2. 프로젝트 소개
시간: 1분

“이 프로젝트는 사용자가 도서를 검색하고, 장바구니에 담고, 주문과 결제를 진행하며, 리뷰와 QnA까지 작성할 수 있는 온라인 서점 웹 서비스입니다.
또한 관리자 페이지를 별도로 구성하여 상품, 회원, 주문, 배송, 공지사항, 리뷰, QnA를 운영할 수 있도록 설계했습니다.

단순한 게시판형 CRUD 프로젝트가 아니라, 실제 서비스 흐름을 기준으로 프론트엔드와 백엔드, 데이터베이스가 하나의 서비스처럼 연결되도록 구현한 것이 핵심입니다.”

3. 개발 배경 및 목적
시간: 1분 30초

“저희는 단순히 기능 몇 개를 만드는 수준이 아니라, 실제 서비스와 가까운 구조를 경험해보고 싶었습니다.
그래서 회원 인증, 상품 관리, 장바구니, 주문, 결제, 배송, 관리자 기능까지 하나의 흐름으로 연결되는 프로젝트를 목표로 삼았습니다.

이 프로젝트를 통해 얻고자 한 핵심 경험은 세 가지입니다.
첫째, 프론트엔드와 백엔드, DB를 분리하면서도 유기적으로 연결하는 경험입니다.
둘째, JWT와 OAuth 같은 인증 구조를 실제로 적용해보는 경험입니다.
셋째, 주문, 결제, 배송처럼 상태 변화가 많은 기능을 일관성 있게 설계하고 구현하는 경험입니다.”

4. 주요 사용자와 요구사항
시간: 1분 30초

“사용자는 크게 비회원, 회원, 관리자 세 가지로 구분됩니다.

비회원은 메인 화면과 상품 목록, 상품 상세 화면까지 접근할 수 있습니다.
회원은 로그인 이후 장바구니, 주문, 결제, 리뷰 작성, QnA 등록, 마이페이지 기능을 사용할 수 있습니다.
관리자는 운영 관점에서 상품과 카테고리, 회원, 주문과 배송, 공지사항, 리뷰, QnA를 관리할 수 있습니다.

이렇게 사용자 역할을 분리한 이유는, 실제 서비스에서 권한에 따라 접근 가능한 기능이 명확하게 나뉘기 때문입니다.
그래서 저희도 사용자와 관리자 기능을 화면 구조와 라우팅, 백엔드 권한 처리 단계에서 분리했습니다.”

5. 기술 스택
시간: 1분 30초

“프론트엔드는 Vue 3 기반 SPA로 구현했습니다.
라우팅은 Vue Router, 상태 관리는 Pinia, API 통신은 Axios를 사용했습니다.

백엔드는 Java 21과 Spring Boot 3를 기반으로 REST API 서버를 구성했습니다.
보안과 인증은 Spring Security와 JWT를 사용했고, 데이터 접근은 JPA를 사용했습니다.

데이터베이스는 PostgreSQL을 사용했습니다.
그리고 프로젝트 설계 문서 기준으로는 Docker Compose와 Nginx 기반 배포 구조를 고려했습니다.

즉, 프론트는 사용자 화면과 인터랙션에 집중하고, 백엔드는 비즈니스 로직과 인증, 데이터 처리를 담당하는 전형적인 분리형 웹 아키텍처를 적용했습니다.”

6. 시스템 아키텍처
시간: 2분

“전체 구조는 Vue 클라이언트가 Spring Boot REST API와 통신하고, 백엔드가 PostgreSQL과 연결되는 형태입니다.

사용자가 프론트엔드에서 로그인하거나 상품 조회, 주문, 결제 같은 요청을 보내면,
프론트엔드는 Axios를 통해 백엔드 API를 호출합니다.
백엔드는 요청을 검증하고 비즈니스 로직을 수행한 뒤, JPA를 통해 DB를 조회하거나 수정합니다.

인증이 필요한 요청은 JWT 토큰 기반으로 처리합니다.
로그인 성공 시 Access Token과 Refresh Token을 발급하고, 이후 보호된 API는 이 토큰을 통해 접근합니다.

또한 관리자 기능은 별도의 관리자 레이아웃과 관리자 API를 통해 분리했고,
Spring Security 권한 검증으로 ADMIN, MANAGER, USER 역할에 따라 접근을 제한했습니다.”

7. 핵심 기능 소개
시간: 2분

“핵심 기능은 크게 다섯 가지로 정리할 수 있습니다.

첫째, 회원 인증 기능입니다.
회원가입, 로그인, JWT 인증, 그리고 Google과 Kakao OAuth 로그인을 지원합니다.

둘째, 상품 조회 기능입니다.
상품 목록, 상세 조회, 검색, 카테고리 필터링을 구현했습니다.

셋째, 장바구니와 주문, 결제 기능입니다.
사용자가 상품을 장바구니에 담고, 주문을 생성한 뒤, 결제를 진행할 수 있습니다.

넷째, 마이페이지 기능입니다.
회원은 자신의 주문 내역, 결제 내역, 리뷰, QnA를 확인하고 관리할 수 있습니다.

다섯째, 관리자 기능입니다.
관리자는 대시보드에서 운영 현황을 확인하고, 상품과 회원, 주문, 배송, 게시판을 관리할 수 있습니다.”

8. 인증 및 권한 처리
시간: 2분

“이 프로젝트에서 중요한 기술 포인트 중 하나는 인증과 권한 처리입니다.

저희는 JWT 기반 Stateless 인증 구조를 적용했습니다.
로그인 시 Access Token과 Refresh Token을 발급하고, 프론트엔드는 Access Token을 이용해 API를 호출합니다.
Access Token이 만료되면 Refresh Token을 통해 재발급을 시도하도록 구성했습니다.

또한 OAuth 로그인도 확장했습니다.
Google OAuth는 Spring Security OAuth2 Client를 활용했고, Kakao OAuth는 별도 API 흐름으로 연동했습니다.

권한 처리 측면에서는 USER, MANAGER, ADMIN 역할을 분리했습니다.
일반 사용자는 사용자 기능만 접근 가능하고, 관리자 페이지는 MANAGER 또는 ADMIN 권한이 있는 경우에만 접근할 수 있도록 설정했습니다.

이 과정을 통해 인증 로직과 권한 검증을 화면 단뿐 아니라 API 단에서도 함께 보호하도록 구현했습니다.”

9. 주문, 결제, 배송 흐름
시간: 2분 30초

“프로젝트에서 가장 핵심적인 비즈니스 흐름은 주문, 결제, 배송입니다.

먼저 사용자가 장바구니에서 상품을 선택하면 주문이 생성됩니다.
이때 주문 정보와 주문 상품 정보가 함께 저장됩니다.

그다음 결제를 진행하면 결제 상태가 갱신됩니다.
결제 성공 시 payment 상태는 PAID로 변경되고, 이후 배송 정보가 생성됩니다.
배송은 READY, SHIPPED, IN_TRANSIT, DELIVERED 같은 상태를 기준으로 관리됩니다.

주문 상태 역시 별도로 관리했습니다.
결제 대기, 결제 완료 후 관리자 확인 대기, 배송 중, 배송 완료, 취소 상태 등을 구분했습니다.

이렇게 주문, 결제, 배송 상태를 각각 분리한 이유는,
실제 서비스에서는 각 흐름이 독립적으로 변할 수 있기 때문입니다.
예를 들어 결제는 완료됐지만 아직 배송은 시작되지 않을 수 있고, 결제 취소나 배송 실패 같은 예외 상황도 발생할 수 있습니다.

그래서 상태 전이를 명확히 나누고, 필요한 곳에서는 트랜잭션을 통해 일관성을 유지하도록 설계했습니다.”

10. 관리자 기능
시간: 2분

“관리자 기능은 실제 운영 업무를 고려해서 구성했습니다.

먼저 대시보드에서는 오늘 주문 수, 신규 회원 수, 오늘과 이번 달 매출, 미답변 QnA 개수, 배송 준비 건수, 재고 부족 상품 수 같은 요약 정보를 볼 수 있습니다.

상품 관리에서는 상품 등록, 수정, 재고 관리, 카테고리 관리가 가능합니다.
회원 관리에서는 회원 목록 조회, 권한 변경, 상태 변경이 가능합니다.
주문과 배송 관리에서는 주문 목록, 승인 대기 주문, 배송 상태 변경을 처리할 수 있습니다.
또한 공지사항, 리뷰, QnA 관리 기능도 제공하여 운영자 입장에서 필요한 콘텐츠 관리 기능도 반영했습니다.

즉, 관리자 페이지는 단순 조회가 아니라 실제 운영 프로세스를 처리하는 화면이라는 점이 중요합니다.”

11. 구현 중 어려움과 해결
시간: 2분 30초

“구현 과정에서 가장 어려웠던 부분은 네 가지였습니다.

첫째는 OAuth 로그인 연동입니다.
일반 로그인과 소셜 로그인을 함께 처리하다 보니 인증 흐름이 꼬이기 쉬웠습니다.
그래서 일반 로그인과 OAuth 로그인 흐름을 분리하고, 최종적으로는 JWT 발급 구조로 통일해서 정리했습니다.

둘째는 관리자와 유저 기능 분리입니다.
화면과 API 모두에서 권한 분리가 필요했기 때문에, 프론트에서는 레이아웃과 라우팅을 나누고, 백엔드에서는 Spring Security 권한 체크를 적용했습니다.

셋째는 다양한 결제 수단 연동입니다.
계좌이체, 토스페이, 카카오페이, 네이버페이를 하나의 구조 안에서 처리해야 했기 때문에, PaymentGateway 인터페이스를 중심으로 PG별 구현체를 분리했습니다.
이를 통해 결제 수단이 늘어나더라도 구조를 확장하기 쉽도록 설계했습니다.

넷째는 프론트엔드와 백엔드의 API 명세 불일치 문제였습니다.
실제로 화면을 붙이는 과정에서 요청 형식과 응답 형식이 맞지 않는 문제가 있었고, 이를 해결하기 위해 DTO를 표준화하고 API 명세를 계속 맞춰가며 통합했습니다.

이 과정을 통해 단순 구현보다 설계와 협업 기준 정리가 훨씬 중요하다는 점을 배울 수 있었습니다.”

12. 결론 및 배운 점
시간: 1분 30초

“정리하면, 이 프로젝트는 단순 쇼핑몰 UI 구현이 아니라 인증, 상품, 장바구니, 주문, 결제, 배송, 리뷰, QnA, 관리자 운영까지 포함한 통합 웹 서비스 프로젝트입니다.

이번 프로젝트를 통해 저희는 세 가지를 크게 배웠습니다.
첫째, 서비스 흐름 설계의 중요성입니다.
둘째, 인증과 권한, 상태 관리, 예외 처리처럼 눈에 잘 안 보이지만 서비스 완성도를 좌우하는 요소들의 중요성입니다.
셋째, 프론트엔드와 백엔드, DB가 하나의 제품처럼 연결되어야 실제 서비스가 된다는 점입니다.

이제 이어서 실제 화면을 통해 사용자 구매 흐름과 관리자 처리 흐름을 시연하겠습니다.”

13. 시연 안내
시간: 30초

“시연에서는 사용자 입장에서 로그인부터 상품 조회, 장바구니, 주문, 결제, 마이페이지까지의 흐름을 먼저 보여드리고,
이후 관리자 페이지로 이동해서 주문 확인, 배송 상태 변경, 운영 화면을 확인하는 순서로 진행하겠습니다.”

14. Q&A 전환 멘트
시간: 10초

“시연까지 마친 뒤 질문을 받겠습니다.”

10분 시연 대본

시연 1. 로그인 및 메인 진입
시간: 1분

“먼저 메인 화면입니다.
비회원 상태에서도 메인과 상품 목록, 상품 상세는 확인할 수 있습니다.
이제 로그인 화면으로 이동해서 일반 로그인 또는 OAuth 로그인 흐름을 보여드리겠습니다.”

“로그인에 성공하면 JWT 기반으로 인증 상태가 유지되고, 이후 인증이 필요한 화면에 접근할 수 있습니다.”

시연 2. 상품 목록 및 상세 조회
시간: 1분 30초

“다음은 상품 목록 화면입니다.
현재 상품 목록을 조회할 수 있고, 검색이나 카테고리 필터를 통해 원하는 상품을 찾을 수 있습니다.”

“상품 하나를 선택하면 상세 화면으로 이동합니다.
여기서 상품 정보와 가격, 설명, 그리고 관련 정보들을 확인할 수 있습니다.”

시연 3. 장바구니 추가
시간: 1분

“이제 상품을 장바구니에 담아보겠습니다.
장바구니에서는 담은 상품 목록, 수량 변경, 선택 여부 변경이 가능하고, 주문할 상품을 선택해서 다음 단계로 진행할 수 있습니다.”

시연 4. 주문 생성
시간: 1분

“장바구니에서 선택한 상품을 기준으로 주문 화면으로 이동하겠습니다.
여기서 주문자 정보와 배송 정보를 확인하고 주문을 생성합니다.”

“이 과정에서 백엔드에서는 주문 정보와 주문 상품 정보가 저장됩니다.”

시연 5. 결제 처리
시간: 2분

“다음은 결제 화면입니다.
현재 프로젝트에서는 계좌이체, 토스페이, 카카오페이, 네이버페이 구조를 지원하도록 구현했습니다.”

“여기서 한 결제 수단을 선택해 결제를 진행하겠습니다.
결제 성공 시 payment 상태가 갱신되고, 이후 완료 화면 또는 성공 화면으로 이동합니다.”

“이 부분에서 중요한 점은 결제 로직이 수단별로 달라도 공통 흐름 안에서 처리되도록 설계했다는 점입니다.”

시연 6. 마이페이지 확인
시간: 1분 30초

“이제 마이페이지로 이동하겠습니다.
여기서는 사용자가 자신의 주문 내역, 결제 내역, 리뷰, QnA를 확인할 수 있습니다.”

“방금 진행한 주문이 반영된 것을 확인할 수 있고, 주문 상세와 결제 상태, 배송 상태도 함께 볼 수 있습니다.”

“필요하면 여기서 리뷰나 QnA 작성 화면도 이어서 보여드릴 수 있습니다.”

시연 7. 관리자 페이지 이동
시간: 1분

“이제 관리자 권한 계정으로 관리자 페이지에 접근하겠습니다.
관리자 페이지는 일반 사용자 페이지와 분리되어 있고, 권한이 없는 경우 접근할 수 없습니다.”

시연 8. 관리자 대시보드
시간: 1분

“관리자 대시보드에서는 주문 현황, 미답변 QnA, 배송 준비 건수, 신규 회원 수, 매출 요약 같은 운영 지표를 볼 수 있습니다.
즉, 단순 관리 기능뿐 아니라 현재 서비스 상태를 빠르게 파악할 수 있도록 구성했습니다.”

시연 9. 주문 및 배송 처리
시간: 1분

“이제 주문 또는 배송 관리 화면으로 이동하겠습니다.
여기서 관리자는 주문을 확인하고, 배송 정보를 입력하거나 배송 상태를 변경할 수 있습니다.”

“사용자가 방금 생성한 주문이 관리자 화면에도 반영되는 것을 확인할 수 있습니다.
즉, 사용자 기능과 관리자 기능이 하나의 흐름으로 연결되어 있다는 점을 보여주는 장면입니다.”
