# ReadMe 기능 구현 발표 정리

작성일: 2026-04-13  
프로젝트: `ReadMe` 온라인 서점  
문서 목적: 구현 완료 기능, 실제 작동 방식, 서비스 방향성을 발표용으로 한 번에 설명하기 위한 정리 문서

## 1. 프로젝트 소개

`ReadMe`는 도서 탐색부터 장바구니, 주문, 결제, 관리자 승인, 배송, 마이페이지 확인까지 이어지는 온라인 서점 서비스입니다.

이번 구현에서는 단순히 화면을 만드는 데 그치지 않고, 실제 서비스 운영 기준에 맞춰 아래 흐름이 자연스럽게 연결되도록 구성했습니다.

- 사용자 구매 흐름
- 관리자 주문 승인 및 배송 운영
- 결제와 환불 처리
- 게시판과 마이페이지 관리

## 2. 핵심 방향성

### 2-1. 사용자 흐름과 관리자 흐름을 분리하되 연결

- 사용자는 상품 탐색, 주문, 결제, 주문 조회를 직관적으로 이용
- 관리자는 주문 승인, 배송 상태 변경, 재고 확인, 문의 대응을 운영 관점에서 관리
- 두 흐름이 상태값과 화면 표시에서 어긋나지 않도록 연결

### 2-2. 결제 완료와 배송 시작을 분리

- 결제 완료 직후 바로 배송으로 넘어가지 않도록 설계
- 결제 완료 후에는 `승인 대기`
- 관리자 승인 후에만 `배송 준비`
- 배송 관리에서 실제 출고 후 `배송 중`, 최종 `배송 완료`

### 2-3. 운영 정책을 실제 기능으로 구현

- 주문 취소와 환불 처리 연결
- 배송 중 취소 시 반품비 차감
- 운송장과 택배사 없이 배송 상태 변경 불가
- 재고 부족 주문만 따로 식별하여 일괄 취소 가능

### 2-4. 화면 문구와 흐름을 일반적인 온라인 서점 형태로 정리

- 과장된 안내 문구 제거
- 리스트형, 게시판형, 모달형 구조 정리
- 텍스트 말줄임, 카드 높이 고정, 행 위치 고정 등 UI 안정성 강화

## 3. 전체 서비스 흐름

### 사용자 구매 흐름

```text
메인 진입
-> 상품 탐색
-> 장바구니
-> 주문서 작성
-> 결제
-> 주문 승인 대기
-> 배송 준비
-> 배송 중
-> 배송 완료
-> 마이페이지 확인
```

### 관리자 운영 흐름

```text
주문 관리
-> 승인 대기 주문 확인
-> 주문 승인
-> 배송 관리에서 배송 건 생성 확인
-> 택배사/운송장 입력
-> 출고 처리
-> 배송 완료 처리
-> 사용자 화면 자동 반영
```

## 4. 상태 설계

### 주문 상태

```text
PAYMENT_PENDING
-> PENDING
-> APPROVAL
-> DELIVERING
-> DELIVERED
```

- `PAYMENT_PENDING`: 결제 대기
- `PENDING`: 결제 완료, 관리자 승인 대기
- `APPROVAL`: 관리자 승인 완료, 배송 준비
- `DELIVERING`: 배송 중
- `DELIVERED`: 배송 완료
- `CANCELED`: 취소

### 핵심 설계 포인트

- 결제 완료 시점에는 배송 레코드를 만들지 않음
- 관리자 승인 시점에만 배송 레코드 생성
- 배송 관리에서 상태를 바꾸면 주문 상태도 같이 변경
- 사용자 화면은 이 상태를 기준으로 배송 준비, 배송 중, 배송 완료를 표시

## 5. 구현 완료 기능

## 5-1. 메인 / 상품 탐색 / 장바구니

### 구현 내용

- 메인 화면 구성
- 상품 목록 조회
- 상품 상세 조회
- 장바구니 담기
- 장바구니 수량 변경 및 삭제
- 장바구니 레이아웃 흔들림 개선

### 작동 방식

1. 사용자가 메인 또는 목록에서 도서를 탐색
2. 상세 화면에서 수량 선택 후 장바구니 추가
3. 장바구니에서 수량 변경 또는 삭제 가능
4. 수량을 변경해도 목록 순서와 레이아웃이 흔들리지 않도록 정렬 기준과 UI 구조를 고정

### 발표 포인트

- 구매 전환의 시작 구간을 안정적으로 구성
- 카드형 영역과 텍스트 영역의 높이를 고정해 화면이 흔들리지 않도록 개선

---

## 5-2. 주문 / 결제

### 구현 내용

- 주문서 작성
- 주문 생성
- 결제 완료 / 실패 / 완료 안내 화면
- 사용자 결제 내역 조회
- 관리자 결제 내역 조회

### 현재 제공 결제 수단

- 계좌이체
- 토스페이
- 카카오페이
- 네이버페이

### 작동 방식

#### 계좌이체

- 다른 간편결제와 분리된 독립 결제 방식으로 구현
- 주문 생성 후 `계좌이체` 결제로 기록
- 결제 내역에서 결제 수단을 별도로 확인 가능

#### 토스페이

- 토스 결제 승인 완료 후 주문 상태를 `PENDING`으로 변경
- 결제 내역에는 `토스페이`로 저장

#### 카카오페이

- 결제 준비 API 호출 후 외부 결제창 이동
- 승인 완료 시 주문 상태와 결제 내역 반영

#### 네이버페이

- 결제 화면과 흐름은 연결되어 있으나, 실결제 정식 연동은 향후 보완 대상

### 발표 포인트

- 결제 수단이 단순 버튼만 다른 것이 아니라 내부 정책도 분리되어 있음
- 결제 완료와 관리자 승인 단계를 분리해 운영 흐름을 명확히 함

---

## 5-3. 주문 승인 / 배송 관리 연동

### 구현 내용

- 주문 승인 기능
- 승인 시 배송 레코드 생성
- 배송 관리 화면 구성
- 출고 처리 버튼 제공
- 배송 상태 변경 시 주문 상태 자동 반영

### 작동 방식

1. 사용자가 결제를 완료하면 주문 상태는 `PENDING`
2. 관리자가 주문 관리에서 승인하면 `APPROVAL`
3. 이 시점에 배송 레코드가 생성되어 배송 관리 목록에 표시
4. 배송 관리에서 택배사와 운송장을 입력
5. `출고 처리`를 누르면 배송 상태가 `배송 중`으로 변경
6. 배송 완료 처리 시 사용자 주문 내역도 자동으로 `배송 완료`로 반영

### 운영 검증 로직

- 택배사와 운송장 번호가 모두 있어야 `배송 중` 또는 `배송 완료`로 변경 가능
- 운송장 없이 출고하는 실수를 방지

### 발표 포인트

- 주문 관리와 배송 관리가 분리되어 있지만 상태는 자동으로 연결
- 실제 운영자가 사용할 수 있는 흐름 중심으로 구현

---

## 5-4. 주문 취소 / 환불 / 반품비 정책

### 구현 내용

- 사용자 주문 취소
- 관리자 취소 시 환불 처리 연계
- 결제 내역에 환불 금액과 반품비 표시
- 사용자/관리자 화면 모두 환불 정보 확인 가능

### 현재 정책

- 교환비: `3,000원`
- 반품비: `6,000원`

### 상태별 처리 방식

- `PAYMENT_PENDING`: 주문 취소만 처리
- `PENDING`, `PAYED`: 전액 환불 후 취소
- `APPROVAL`: 출고 전 주문으로 보고 전액 환불 후 취소
- `DELIVERING`: 반품비 6,000원 차감 후 환불
- `DELIVERED`: 주문 취소 불가

### 핵심 구현 포인트

- 환불 여부를 단순 주문 상태가 아니라 실제 결제 상태 기준으로 판단
- `PENDING` 상태라도 결제가 완료된 주문이면 정상 환불 처리
- 반품비, 실환불금액, 취소 정보를 저장하고 화면에 노출

### 발표 포인트

- 단순 상태 변경이 아니라 실제 환불 정책까지 시스템에 반영

---

## 5-5. 관리자 주문 관리

### 구현 내용

- 주문번호, 주문회원, 주문 도서 표시
- 상태별 주문 조회
- 체크박스 선택
- 일괄 승인
- 재고 부족 주문 식별
- 재고 부족 주문만 일괄 취소

### 재고 부족 기준

```text
현재 재고 < 주문 수량
```

예시:

- 재고 0, 주문 수량 1 -> 재고 부족
- 재고 1, 주문 수량 2 -> 재고 부족
- 재고 3, 주문 수량 2 -> 정상

### 작동 방식

1. 관리자가 주문 목록에서 주문 상태 확인
2. `재고 부족 상품 주문만 보기`로 필터링
3. 재고 부족 배지가 붙은 주문만 선택
4. `선택 주문 재고 부족 취소` 실행
5. 결제 완료 주문은 환불까지 함께 처리

### 발표 포인트

- 일반 일괄 취소가 아니라 품절/재고부족 상황만 따로 분리
- 운영 사고를 줄이는 방향으로 관리자 UX 구성

---

## 5-6. 배송 관리

### 구현 내용

- 배송 ID
- 주문번호
- 주문회원
- 택배사
- 운송장 번호
- 상태
- 출고일
- 수정 기능

### 작동 방식

1. 주문 승인 후 배송 건이 생성됨
2. 배송 관리에서 배송 정보를 입력
3. `출고 처리`로 배송 준비에서 배송 중으로 전환
4. 배송 완료 처리 시 사용자 주문 상태도 자동 반영

### 발표 포인트

- 관리자 입장에서 배송 상태를 실질적으로 운영할 수 있는 화면
- 주문 화면과 배송 화면이 따로 놀지 않음

---

## 5-7. 마이페이지

### 구현 내용

- 주문 내역
- 주문 상세
- 결제 내역
- 내 문의
- 내 리뷰
- 회원정보 수정
- 비밀번호 변경
- 회원 탈퇴

### 작동 방식

- 주문 상태는 관리자 승인과 배송 상태에 따라 자동 반영
- 결제 취소 시 환불 금액, 반품비 정보도 함께 표시
- `내 문의`, `내 리뷰`는 리스트형으로 정리하고 상세와 수정은 모달로 처리

### 발표 포인트

- 구매 이후의 전체 경험을 한 곳에서 확인할 수 있도록 구성

---

## 5-8. 게시판

### 구현 내용

- 공지사항
- QnA
- 리뷰

### 현재 화면 방향

#### 공지사항

- 게시판형 리스트
- 제목 클릭 시 모달 상세 보기

#### QnA

- 게시판형 리스트
- 공개 문의 / 내 문의 구분
- 작성 / 상세 / 수정 / 삭제를 모달 흐름으로 처리

#### 리뷰

- 전체 리뷰 페이지 운영
- 마이페이지 리뷰는 리스트 + 모달 구조로 정리

### 발표 포인트

- 카드형, 상세 분할형보다 일반 게시판 구조에 가깝게 정리
- 사용자가 익숙한 커뮤니티/고객센터 형태로 맞춤

---

## 5-9. 문구 및 UI 안정화

### 구현 내용

- AI스럽거나 과장된 문구 제거
- 일반 온라인 서점 톤으로 텍스트 정리
- 푸터 가운데 정렬
- 카드 높이와 위치 고정
- 긴 제목과 설명 말줄임 처리

### 발표 포인트

- 디자인을 바꾸지 않고도 서비스 완성도를 높인 작업
- 실제 사용 시 더 안정적이고 자연스러운 화면 경험 제공

## 6. 실제 시연 순서

### 시연 1. 사용자 구매 흐름

1. 메인 진입
2. 상품 목록 또는 상세 이동
3. 장바구니 담기
4. 주문서 작성
5. 결제 수단 선택
6. 결제 완료
7. 마이페이지 주문/결제 내역 확인

### 시연 2. 관리자 주문 및 배송 흐름

1. 관리자 로그인
2. 주문 관리에서 승인 대기 주문 확인
3. 주문 승인
4. 배송 관리로 이동
5. 택배사 / 운송장 입력
6. 출고 처리
7. 사용자 화면에서 배송 중 반영 확인
8. 배송 완료 처리 후 사용자 화면에서 배송 완료 확인

### 시연 3. 환불 정책

1. 배송 준비 주문 취소
2. 전액 환불 확인
3. 배송 중 주문 취소
4. 반품비 6,000원 차감 확인
5. 결제 내역에서 실환불금액 확인

### 시연 4. 게시판 및 마이페이지

1. 공지사항 리스트 확인
2. 공지사항 모달 상세 확인
3. QnA 작성
4. QnA 수정 및 답변 확인
5. 내 리뷰 리스트와 모달 상세 확인

## 7. 이번 구현의 강점

### 7-1. 운영 단계가 명확함

- 결제
- 승인
- 배송 준비
- 배송 중
- 배송 완료

단계가 섞이지 않고 분리되어 있습니다.

### 7-2. 사용자 화면과 관리자 화면이 연결됨

- 관리자 상태 변경이 사용자 주문 내역에 즉시 반영
- 배송 관리와 주문 상태가 따로 놀지 않음

### 7-3. 실제 운영 정책을 기능으로 반영함

- 반품비
- 환불 금액
- 재고 부족 주문
- 택배사/운송장 검증

등을 화면과 로직에 모두 반영했습니다.

### 7-4. 사용 흐름 중심의 UI 정리

- 게시판형 리스트
- 모달 상세
- 카드 높이 고정
- 장바구니 행 위치 고정
- 말줄임 처리

등 실제 사용성 기준의 개선을 반영했습니다.

## 8. 현재 남은 과제

### 8-1. 네이버페이 실결제 정식 연동

- 현재는 테스트/샌드박스 성격이 강함
- 실서비스용 연동 마무리 필요

### 8-2. 관리자 일괄 상태 변경 UX 확장

- 현재는 일괄 승인과 재고 부족 주문 일괄 취소 중심
- 향후 배송 완료 등 범용 일괄 처리 UX 확장 가능

### 8-3. 관리자 주문 승인 화면 통합

- 주문 관리와 승인 화면의 일부 역할이 중첩됨

### 8-4. 메인 사이드바 카테고리 API 연동

- 관리자 카테고리 변경이 사용자 사이드바에 자동 반영되도록 보완 필요

### 8-5. 비밀번호 재설정 보안 강화

- 실서비스 수준의 인증 절차 보완 필요

## 9. 발표 마무리 멘트 예시

이번 프로젝트는 단순한 서점 화면 구현이 아니라,
실제 온라인 서점 운영 흐름을 기준으로 사용자 주문, 관리자 승인, 배송 처리, 환불 정책, 게시판 운영까지 연결하는 데 중점을 두었습니다.

특히 결제 완료와 배송 시작을 분리하고, 관리자 화면과 사용자 화면의 상태를 맞물리게 설계한 점이 핵심 구현 포인트입니다.

앞으로는 네이버페이 실결제 연동, 일괄 운영 기능 확장, 카테고리 동적 연동 등을 보완하여 서비스 완성도를 더 높일 계획입니다.
