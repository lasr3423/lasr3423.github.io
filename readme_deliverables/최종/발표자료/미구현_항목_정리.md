## 목적

이 문서는 현재 프로젝트에서 **아직 구현이 완전히 마무리되지 않은 항목만 따로 정리한 문서**입니다.

정리 기준은 아래와 같습니다.

- 화면은 있으나 실제 운영 흐름이 완결되지 않은 기능
- 테스트/샌드박스 수준에 머물러 있는 기능
- 관리자 운영 관점에서 추가 정리가 필요한 기능
- 실서비스 보안/운영 기준으로 아직 보완이 필요한 기능

## 현재 기준 요약

- 프론트엔드 빌드: `npm run build` 통과
- 백엔드 컴파일: `bash ./gradlew compileJava` 통과
- 주문 취소/환불, 승인 후 배송 생성, 배송 상태 동기화, 게시판형 리스트 전환 등 핵심 흐름은 반영 완료
- 다만 아래 항목들은 아직 미구현 또는 부분 구현 상태입니다.

## 미구현 항목 요약표

## 1. 네이버페이 실결제 정식 연동

### 현재 상태

- 결제 화면에는 네이버페이 선택지가 존재합니다.
- 하지만 현재 구현은 테스트/샌드박스 흐름을 기준으로 연결되어 있습니다.
- 실제 결제 준비, 승인, 취소 흐름이 운영 스펙 수준으로 완전히 정리된 상태는 아닙니다.

### 문제점

- 운영 환경에 바로 적용하기 어렵습니다.
- 실결제 검증과 오류 대응 시나리오가 부족합니다.
- 네이버 공식 스펙 기준의 준비/승인/취소 파라미터 정리가 더 필요합니다.

### 정리 방향

- `ready`, `approve`, `cancel` 흐름을 네이버 공식 스펙 기준으로 재정리
- 샌드박스와 운영 환경 설정 분리
- 프론트 결제 시작 UX와 결과 처리 메시지 정리

## 2. 관리자 주문 일괄 상태 변경 UX 확장

### 현재 상태

- 백엔드에는 주문 일괄 상태 변경 API가 존재합니다.
- 프론트 주문 관리 화면에서는 현재 `일괄 승인`과 `재고 부족 주문 일괄 취소` 중심으로 구성되어 있습니다.

### 이미 구현된 부분

- 승인 대기 주문 일괄 승인
- 재고 부족 주문 필터
- 재고 부족 주문만 선택해서 일괄 취소

### 아직 남은 부분

- 상태별 범용 일괄 액션 구성
- 예를 들어 아래 같은 흐름은 아직 UI로 일반화되어 있지 않습니다.
  - `APPROVAL -> DELIVERING`
  - `DELIVERING -> DELIVERED`
  - 운영 사유별 일괄 처리 버튼 정리

### 정리 방향

- 선택된 주문 상태에 따라 가능한 일괄 액션만 보여주기
- 실수 방지를 위해 상태별 허용 액션을 분리
- 배송 단계 일괄 처리와 취소 사유 기반 액션을 구분

## 3. 관리자 주문 승인 화면 통합

### 현재 상태

- 주문 관리 화면에서 승인 관련 기능을 이미 수행할 수 있습니다.
- 그런데 별도로 `/admin/order/approval` 화면도 남아 있습니다.

### 문제점

- 비슷한 역할의 화면이 두 개 존재합니다.
- 운영자가 어느 화면을 기준으로 써야 하는지 혼선이 생길 수 있습니다.
- 유지보수 시 같은 기능을 두 화면에서 같이 관리해야 합니다.

### 정리 방향

- 주문 관리 화면으로 승인 기능을 통합
- 기존 승인 전용 화면은 제거하거나 진입 경로만 남기기
- 승인 대기 목록 기준과 상태 라벨을 한 화면 기준으로 통일

## 4. 유저 메인 사이드바 카테고리 API 연동

### 현재 상태

- 관리자 카테고리 관리는 API 기반으로 동작합니다.
- 하지만 유저 메인 사이드바는 아직 관리자 변경 사항이 자동 반영되는 구조로 완전히 연결되지 않았습니다.

### 문제점

- 관리자가 카테고리를 수정해도 메인 사이드바 메뉴에 바로 반영되지 않을 수 있습니다.
- 운영 데이터와 사용자 화면 메뉴가 어긋날 수 있습니다.

### 정리 방향

- 메인 사이드바가 `/api/category/...` 계열 응답을 사용하도록 연결
- 대분류/소분류를 실제 카테고리 데이터 기준으로 렌더링
- 카테고리 비활성화 여부까지 반영

## 5. 비밀번호 재설정 보안 강화

### 현재 상태

- 현재 비밀번호 재설정은 이메일 인증이나 토큰 검증 없이 동작합니다.
- 즉, 기능은 동작하지만 실서비스 기준으로는 임시 구현에 가깝습니다.

### 문제점

- 보안상 취약합니다.
- 실제 운영 서비스에서는 인증 없는 재설정은 위험합니다.

### 정리 방향

- 이메일 인증 코드 또는 토큰 기반 재설정 도입
- 재설정 링크 만료 시간 적용
- 인증 성공 후에만 새 비밀번호 입력 허용
