아래 대본은 업로드된 15장 PPT 기준이다.

### 슬라이드 1. 표지

안녕하세요. 저희는 Vue 3, Spring Boot, PostgreSQL 기반으로 온라인 서점 웹 서비스를 구현한 `README` 팀입니다.  
이번 프로젝트는 사용자의 구매 흐름과 관리자의 운영 흐름을 하나의 서비스로 연결하는 것을 목표로 진행했습니다.  
발표는 프로젝트 개요부터 설계 산출물, 핵심 구현 내용, 그리고 마지막 시연 순서로 진행하겠습니다.

### 슬라이드 2. 프로젝트 소개

저희 프로젝트는 도서 검색, 주문, 결제, 배송, 리뷰, QnA 기능을 통합한 온라인 서점 서비스입니다.  
일반 사용자와 관리자 페이지를 분리해서, 사용자 입장에서는 구매 흐름이 자연스럽게 이어지고, 관리자 입장에서는 주문과 배송, 회원, 게시판 운영이 가능하도록 구성했습니다.  
개발 기간은 2026년 3월 13일부터 4월 12일까지였습니다.

### 슬라이드 3. 개발 배경 및 목적

이번 프로젝트의 목적은 단순 CRUD 구현이 아니라, 실제 서비스에 가까운 커머스 구조를 경험해보는 것이었습니다.  
그래서 회원 인증, 상품 관리, 장바구니, 주문, 결제, 배송, 관리자 기능까지 하나의 흐름으로 연결되는 구조를 목표로 잡았습니다.  
특히 프론트엔드, 백엔드, DB를 분리하되 실제로 유기적으로 연동하는 경험을 얻고자 했습니다.

### 슬라이드 4. 주요 사용자와 요구사항

요구사항 단계에서는 사용자를 비회원, 회원, 관리자 세 그룹으로 나눠서 정리했습니다.  
비회원은 상품 조회 중심, 회원은 구매와 마이페이지 기능 중심, 관리자는 운영 기능 중심으로 역할을 구분했습니다.  
이 기준에 맞춰 화면과 API 접근 권한도 분리했습니다.

### 슬라이드 5. 기술 스택

프론트엔드는 Vue 3, Vue Router, Pinia, Axios를 사용했고, 백엔드는 Java 21과 Spring Boot 3, JPA, Spring Security를 사용했습니다.  
데이터베이스는 PostgreSQL이고, 실행 구조는 Docker Compose와 Nginx를 고려해 설계했습니다.  
인증은 JWT와 OAuth, 결제는 PG 확장 구조를 반영했습니다.

### 슬라이드 6. 시스템 아키텍처

전체 구조는 Vue SPA가 Spring Boot REST API와 통신하고, 백엔드가 PostgreSQL을 사용하는 3계층 구조입니다.  
인증이 필요한 요청은 JWT 토큰 검증을 거치고, 관리자 API는 별도 권한 검사를 수행합니다.  
즉, 화면, 비즈니스 로직, 데이터 저장 계층을 분리해 유지보수성과 확장성을 고려했습니다.

### 슬라이드 7. 핵심 기능 소개

핵심 기능은 회원 인증, 상품 조회, 장바구니와 주문/결제, 마이페이지, 관리자 기능으로 정리할 수 있습니다.  
특히 이번 프로젝트는 로그인부터 주문, 결제, 관리자 승인, 배송 상태 반영까지 연결되는 점이 핵심입니다.  
단순 화면 구현이 아니라 상태 변화가 이어지는 흐름을 구현했다는 점이 중요합니다.

### 슬라이드 8. 인증 및 권한 처리

인증은 JWT 기반 Stateless 구조로 구현했습니다.  
로그인 시 Access Token과 Refresh Token을 발급하고, Spring Security를 통해 보호가 필요한 API에 접근 제어를 적용했습니다.  
또한 Google OAuth와 Kakao OAuth 흐름도 반영해서 인증 확장 구조를 만들었습니다.

### 슬라이드 9. 주문, 결제, 배송 흐름

이 슬라이드는 멘트를 반드시 실제 구현 기준으로 보정해서 말해야 합니다.

추천 멘트:

사용자가 장바구니에서 상품을 선택하면 주문이 먼저 생성되고, 결제가 완료되면 주문 상태는 `PENDING`, 즉 관리자 승인 대기 상태가 됩니다.  
그 다음 관리자가 주문을 승인하면 그 시점에 배송 레코드가 생성되고, 이후 배송 관리 화면에서 택배사와 운송장을 입력한 뒤 출고 처리와 배송 완료 처리를 진행합니다.  
즉, 저희는 결제 완료와 배송 시작을 분리해서 운영 단계가 섞이지 않도록 구현했습니다.

### 슬라이드 10. 관리자 기능

이 슬라이드도 실제 구현 범위 기준으로 말하는 것이 중요합니다.

추천 멘트:

관리자 페이지에서는 대시보드, 상품 및 카테고리 관리, 회원 관리, 주문 관리, 배송 관리, 결제 내역 조회, 공지사항, 리뷰, QnA 관리 기능을 제공합니다.  
특히 주문 승인 후 배송 생성, 재고 부족 주문 식별, 배송 상태 변경 시 사용자 주문 상태 반영 같은 운영 중심 기능을 구현했습니다.  
슬라이드에 보이는 백업, 로그, 시스템 설정 같은 항목은 구현 완료 기능으로 말하지 않고, 운영 확장 방향으로만 짧게 언급하는 것이 안전합니다.

### 슬라이드 11. 커뮤니티 기능 요약

커뮤니티 기능은 공지사항, 리뷰, QnA 세 영역으로 구성했습니다.  
공지사항은 관리자 작성과 사용자 조회 흐름, 리뷰는 구매 기반 작성과 이미지 첨부, QnA는 질문과 관리자 답변 구조를 반영했습니다.  
여기서는 실시간이라는 표현보다는, 관리자 답변이 반영되는 구조라고 표현하는 것이 더 정확합니다.

### 슬라이드 12. 구현 중 어려움과 해결

어려웠던 부분은 인증 구조 정리, 관리자와 사용자 기능 분리, 여러 결제 수단을 공통 구조로 묶는 일, 그리고 프론트와 백엔드 API 명세를 맞추는 작업이었습니다.  
이를 해결하기 위해 JWT와 OAuth 구조를 정리하고, PaymentGateway 인터페이스로 PG사를 분리하고, DTO와 상태값을 통일하는 방식으로 정리했습니다.

### 슬라이드 13. 결론 및 배운 점

이번 프로젝트를 통해 화면 구현보다 더 중요한 것이 서비스 흐름 설계라는 점을 배웠습니다.  
특히 인증, 권한, 주문 상태, 결제 상태, 배송 상태처럼 눈에 잘 안 보이는 부분이 서비스 완성도를 좌우한다는 점을 체감했습니다.  
앞으로는 결제 고도화와 운영 기능 보완을 통해 완성도를 더 높일 수 있습니다.

시연 순서는 로그인, 상품 조회, 장바구니, 주문/결제, 마이페이지, 관리자 페이지 순서로 진행하겠습니다.

## 5. 시연 대본

### 시연 시작 전 멘트

지금부터는 실제 구현된 화면 기준으로 사용자 구매 흐름과 관리자 처리 흐름을 시연하겠습니다.

### 시연 1. 사용자 로그인

- 비회원도 메인과 상품 조회는 가능하다는 점 설명
- 로그인 후 인증이 필요한 기능 접근 가능하다고 설명

멘트:

먼저 비회원 상태에서는 메인과 상품 조회만 가능합니다.  
이제 로그인해서 장바구니와 주문, 마이페이지 기능까지 이어서 보여드리겠습니다.

### 시연 2. 상품 조회와 장바구니

멘트:

상품 목록에서 원하는 도서를 선택하고 상세 화면으로 들어갈 수 있습니다.  
여기서 수량을 선택해 장바구니에 담고, 장바구니에서는 수량 변경과 삭제가 가능합니다.

### 시연 3. 주문과 결제

멘트:

장바구니에서 선택한 상품으로 주문을 생성하고 결제를 진행하겠습니다.  
저희는 계좌이체, 토스페이, 카카오페이, 네이버페이 구조를 반영했고, 결제가 완료되면 주문은 바로 배송이 아니라 `관리자 승인 대기` 상태로 들어갑니다.

### 시연 4. 마이페이지

멘트:

마이페이지에서는 주문 내역, 결제 내역, 리뷰, QnA를 확인할 수 있습니다.  
방금 진행한 주문이 반영된 것을 확인할 수 있고, 이후 관리자 처리에 따라 상태가 바뀌는 것도 확인할 수 있습니다.

### 시연 5. 관리자 주문 승인

멘트:

이제 관리자 페이지로 이동하겠습니다.  
관리자는 승인 대기 주문을 확인하고 승인할 수 있습니다.  
이 승인 시점에 배송 레코드가 생성된다는 점이 저희 구현의 핵심 포인트입니다.

### 시연 6. 배송 관리

멘트:

배송 관리에서는 택배사와 운송장을 입력한 뒤 출고 처리와 배송 완료 처리를 할 수 있습니다.  
그리고 이 상태 변경은 사용자 주문 내역에도 자동으로 반영됩니다.

### 시연 7. 마무리

멘트:

즉, 사용자 화면과 관리자 화면이 분리되어 있지만 주문 상태와 배송 상태는 하나의 흐름으로 연결되어 있다는 점을 확인할 수 있습니다.

## 6. 발표 때 꼭 강조할 것

- 이 프로젝트는 단순 쇼핑몰 화면 구현이 아니라 `사용자 흐름`과 `관리자 운영 흐름`을 함께 구현한 프로젝트라는 점
- JWT 기반 인증과 역할 기반 권한 분리를 적용했다는 점
- 주문, 결제, 배송 상태를 분리해서 관리했다는 점
- 결제 완료 후 바로 배송이 아니라 `관리자 승인 후 배송 생성`으로 운영 정책을 반영했다는 점
- 환불/취소 정책과 반품비 차감까지 시스템 로직에 반영했다는 점
- 관리자 상태 변경이 사용자 마이페이지에 연결된다는 점

---

## 7. 발표 때 과장하면 안 되는 것

- `결제 완료 즉시 배송 생성`이라고 말하면 안 됨
- `백업`, `로그 모니터링`, `시스템 설정`이 구현 완료라고 말하면 안 됨
- `네이버페이 실결제 완전 연동 완료`라고 말하면 안 됨
- `실시간 QnA 답변 시스템`이라고 강하게 말하기보다 `관리자 답변 구조 구현`이라고 말하는 것이 안전

## 8. 질문받았을 때 답변 예시

### Q. 프로젝트에서 가장 핵심 기능은 무엇인가요?

가장 핵심은 주문, 결제, 배송, 관리자 승인 흐름을 하나의 서비스 정책으로 연결한 부분입니다.  
특히 결제 완료와 배송 시작을 분리해서 실제 운영 기준에 맞춘 점이 핵심입니다.

### Q. 어려웠던 점은 무엇인가요?

인증 구조와 결제 구조, 그리고 프론트와 백엔드 상태값을 맞추는 것이 가장 어려웠습니다.  
그래서 JWT/OAuth 구조를 정리하고, 결제는 Gateway 구조로 분리하고, 상태 전이 규칙을 문서화해서 해결했습니다.

### Q. 아직 보완할 점은 무엇인가요?

네이버페이 실결제 정식 연동, 관리자 일괄 처리 UX 확장, 메인 카테고리 연동, 비밀번호 재설정 보안 강화가 남아 있습니다.
