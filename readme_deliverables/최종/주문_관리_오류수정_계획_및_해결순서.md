# 주문 관리 오류 수정 계획 및 해결 순서

## 1. 문서 목적

- 관리자 주문 관리 화면에서 발생한 상태 변경 오류의 원인을 정리한다.
- 결제 확인, 배송 승인, 주문 취소 흐름을 운영 기준에 맞게 다시 정의한다.
- 단건 처리와 일괄 처리 기능을 어떤 순서로 수정해야 하는지 구현 순서대로 정리한다.

## 2. 현재 확인된 핵심 문제

### 2.1 주문 상태 의미가 화면과 백엔드에서 다름

- 백엔드 기준 상태 의미
- `PAYMENT_PENDING`: 결제 미확인
- `PENDING`: 결제 확인 완료, 관리자 승인 대기
- `PAYED`: 레거시 결제 완료 상태, 신규 기준에서는 `PENDING`과 유사
- `APPROVAL`: 배송 승인 완료
- `CANCELED`: 주문 취소

- 현재 관리자 주문 목록 화면에서는 `PENDING`을 결제대기처럼 보여주고 있어 운영자가 상태를 잘못 변경하게 된다.
- 이 때문에 이미 결제 확인이 끝난 주문에 대해 다시 `결제완료` 전이를 시도하는 문제가 발생한다.

### 2.2 승인되지 않는 직접 원인

- 백엔드 `OrderService.validateAdminStatusTransition()`에서는 `PAYMENT_PENDING` 상태 주문은 취소만 허용한다.
- `PENDING` 또는 `PAYED` 상태 주문만 `APPROVAL` 또는 `CANCELED`로 바꿀 수 있다.
- 따라서 화면에서 `PAYMENT_PENDING -> PAYED` 또는 `PENDING -> PAYED` 같은 전이를 시도하면 예외가 발생한다.

### 2.3 운영 기능이 화면에 제대로 반영되지 않음

- 관리자가 직접 결제를 확인한 뒤 승인 대기 상태로 올리는 기능이 필요하다.
- 여러 주문을 한 번에 승인하거나 취소할 수 있는 체크박스 기반 일괄 처리 기능이 필요하다.
- 현재 `/admin/order/approval` 화면은 `PAYED` 기준에 치우쳐 있어 신규 상태 체계와 맞지 않는다.

## 3. 목표 운영 흐름

### 3.1 최종 주문 처리 흐름

1. 주문 생성
2. `PAYMENT_PENDING`
3. 관리자 결제 확인
4. `PENDING`
5. 관리자 배송 승인
6. `APPROVAL`
7. 필요 시 배송 진행

### 3.2 취소 처리 흐름

- `PAYMENT_PENDING`, `PENDING`, `PAYED` 상태에서는 관리자 취소가 가능해야 한다.
- PG 결제 주문은 기존 결제 취소 로직과 연결되어야 한다.
- 수동 결제 확인 주문은 PG 호출 없이 내부 결제 상태만 취소 처리해야 한다.

## 4. 오류 수정 계획

### 4.1 1차 수정: 상태 의미와 UI 표시 정렬

- 관리자 주문 목록 화면의 상태 라벨을 백엔드 의미와 동일하게 변경한다.
- 드롭다운의 표시 문구도 운영 기준에 맞게 정리한다.
- `PENDING`은 `결제 확인 완료 / 승인 대기`로 보여준다.
- `PAYMENT_PENDING`은 `결제 미확인`으로 보여준다.
- `PAYED`는 레거시 데이터용 `결제 확인 완료 / 승인 대기`로 보여준다.

### 4.2 2차 수정: 결제 확인 기능 추가

- 관리자가 직접 결제 확인 시 `PAYMENT_PENDING -> PENDING` 전이를 허용한다.
- 이 동작은 단순 상태 변경이 아니라 결제 상태도 함께 맞춰야 한다.
- `Payment` 레코드가 있으면 `PAID` 상태로 갱신한다.
- `Payment` 레코드가 없으면 수동 결제 기록을 생성한다.
- 수동 결제 생성 기본값
- `paymentProvider = MANUAL`
- `method = ADMIN_CONFIRMED`
- `paymentStatus = PAID`
- `amount = order.finalPrice`
- `paidAt = now`

### 4.3 3차 수정: 배송 승인 기능 정리

- `PENDING` 또는 `PAYED` 상태 주문만 `APPROVAL`로 변경 가능하게 유지한다.
- `APPROVAL` 전환 시 배송 레코드가 없으면 `READY` 상태로 생성한다.
- 이미 배송 레코드가 있으면 중복 생성하지 않는다.

### 4.4 4차 수정: 관리자 취소 기능 정리

- `PAYMENT_PENDING`, `PENDING`, `PAYED` 상태 주문은 관리자 취소 가능 대상으로 본다.
- PG 결제 건이면 기존 PG 취소 로직을 호출한다.
- 수동 결제 건이면 외부 PG 취소 호출 없이 `PaymentStatus.CANCELLED`만 반영한다.
- `APPROVAL`, `CANCELED` 상태는 변경 불가 상태로 유지한다.

### 4.5 5차 수정: 일괄 처리 기능 추가

- `/admin/order/list` 화면에 체크박스를 추가한다.
- 전체 선택, 개별 선택, 선택 개수 표시를 제공한다.
- 선택한 주문들에 대해 아래 기능을 제공한다.
- 일괄 결제 확인
- 일괄 배송 승인
- 일괄 취소

- 버튼 활성화 조건
- `일괄 결제 확인`: 선택 주문이 모두 `PAYMENT_PENDING`
- `일괄 배송 승인`: 선택 주문이 모두 `PENDING` 또는 `PAYED`
- `일괄 취소`: 선택 주문이 모두 `PAYMENT_PENDING`, `PENDING`, `PAYED`

## 5. 해결 방법 구현 순서

### Step 1. 백엔드 상태 전이 규칙 정리

- `OrderService.updateOrderStatus()`를 공통 상태 전이 진입점으로 유지한다.
- `validateAdminStatusTransition()`를 운영 정책에 맞게 재정비한다.
- `PAYMENT_PENDING -> PENDING` 전이를 신규 허용 규칙으로 추가한다.
- `PENDING/PAYED -> APPROVAL`, `PENDING/PAYED -> CANCELED` 규칙을 유지한다.

### Step 2. 수동 결제 확인 처리 구현

- 주문 상태를 `PENDING`으로 바꿀 때 결제 상태 동기화 로직을 추가한다.
- 별도 내부 메서드로 `confirmManualPayment(order)` 형태의 처리 흐름을 만든다.
- 결제 정보가 없는 주문도 관리자 확인으로 정상 처리할 수 있게 한다.
- 필요하면 `PaymentProvider`에 `MANUAL` 값을 추가한다.

### Step 3. 관리자 취소 처리 보강

- 관리자 상태 변경에서 `CANCELED` 전환 시 결제 취소 분기 로직을 탄다.
- PG 결제 주문과 수동 결제 주문을 구분한다.
- 취소 사유는 기본값을 두고 필요 시 일괄 API에서 전달 가능하게 한다.

### Step 4. 일괄 상태 변경 API 추가

- 신규 API 예시

```json
PATCH /api/admin/orders/status/bulk
```

- 요청 예시

```json
{
  "orderIds": [1, 2, 3],
  "targetStatus": "PENDING",
  "reason": "관리자 일괄 처리"
}
```

- 응답은 성공/실패 건수를 함께 내려준다.
- 일부 주문만 실패하는 경우에도 전체 결과를 확인할 수 있게 주문별 메시지를 포함한다.

### Step 5. 관리자 주문 목록 UI 수정

- 상태 컬럼 라벨을 수정한다.
- 현재의 단순 상태 변경 드롭다운은 운영 액션 중심으로 재구성한다.
- 상태에 따라 유효한 액션만 노출한다.
- 예시
- `PAYMENT_PENDING`: 결제 확인, 취소
- `PENDING/PAYED`: 배송 승인, 취소
- `APPROVAL/CANCELED`: 비활성 또는 변경 불가 표시

### Step 6. 체크박스 및 일괄 처리 UI 추가

- 표 좌측에 체크박스 컬럼을 추가한다.
- 헤더에 전체 선택 체크박스를 둔다.
- 상단 툴바에 선택 개수와 일괄 액션 버튼을 표시한다.
- 유효하지 않은 상태 조합이 섞이면 버튼을 비활성화하거나 안내 메시지를 보여준다.

### Step 7. 승인 요청 화면 정리

- `/admin/order/approval` 화면은 별도 처리 화면이 아니라 `/admin/order/list`의 승인 대기 필터 진입점으로 단순화한다.
- 승인 대기 대상은 `PENDING + PAYED`를 함께 본다.

### Step 8. 대시보드 집계 정리

- 미승인 주문 수는 `PENDING + PAYED` 합산으로 표시한다.
- 운영자가 실제 승인 대기 건수를 혼동하지 않도록 문구를 조정한다.

## 7. 검증 항목

### 단건 처리

- `PAYMENT_PENDING -> PENDING` 성공 여부
- `PENDING -> APPROVAL` 성공 여부
- `PAYED -> APPROVAL` 레거시 호환 여부
- `PAYMENT_PENDING -> PAYED` 금지 여부
- `APPROVAL -> 변경 시도` 차단 여부
- `CANCELED -> 변경 시도` 차단 여부

### 결제 연동

- 수동 결제 확인 시 `Payment` 생성 또는 갱신 여부
- PG 결제 취소 시 외부 취소 호출 여부
- 수동 결제 취소 시 내부 상태만 취소되는지 여부

### 일괄 처리

- 전체 선택 / 개별 선택 정상 동작
- 선택 개수 정확성
- 일괄 결제 확인 버튼 활성화 조건
- 일괄 배송 승인 버튼 활성화 조건
- 일괄 취소 버튼 활성화 조건
- 일부 실패 시 주문별 결과 메시지 표시 여부

### 빌드 및 테스트

- 프론트 `npm run build`
- 백엔드 주문/결제 관련 테스트
- 주문 관리 화면 수동 시나리오 점검

## 8. 결론

- 이번 오류의 본질은 단순 예외 처리 문제가 아니라 주문 상태 체계와 관리자 화면 표현이 어긋난 데서 발생한 운영 오류다.
- 따라서 해결은 화면 문구 수정만으로 끝내면 안 되고, 결제 확인, 승인, 취소, 일괄 처리까지 포함해 주문 관리 흐름 전체를 재정렬해야 한다.
- 우선순위는 `상태 의미 정리 -> 결제 확인 기능 추가 -> 취소 로직 보강 -> 일괄 처리 추가 -> 대시보드/승인 화면 정리` 순서로 진행하는 것이 가장 안전하다.
